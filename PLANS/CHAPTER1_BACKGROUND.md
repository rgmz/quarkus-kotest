# Why Kotest cannot work with Quarkus — and what it would take to fix

Kotest and Quarkus are fundamentally incompatible because **Kotest runs as a separate JUnit Platform test engine that never triggers Quarkus's JUnit Jupiter–specific bootstrap chain**. The `@QuarkusTest` annotation activates `QuarkusTestExtension` through Jupiter's `@ExtendWith` mechanism, but Kotest's `KotestJunitPlatformTestEngine` is a peer engine to Jupiter — not a Jupiter extension — so it bypasses Jupiter's extension lifecycle entirely. CDI injection, classloader setup, and application startup never occur. No working integration exists as of April 2026, the official example repository was archived in February 2025, and the Quarkus tracking issue (#19749) remains open with no assignee. The most practical path forward would require either Quarkus abstracting its test infrastructure away from Jupiter (as Spring did with `TestContextManager`) or building a dedicated Kotest extension that reimplements Quarkus's bootstrap logic — a substantial engineering effort that nobody has yet completed.

## The architecture mismatch runs deeper than annotations

The common misconception is that since Kotest "runs on JUnit 5," it should work with JUnit 5 extensions. This confuses the **JUnit Platform** (a launcher and discovery API) with **JUnit Jupiter** (a specific test engine with its own extension model). Kotest implements `org.junit.platform.engine.TestEngine` with engine ID `"kotest"`, making it a sibling to Jupiter under the Platform umbrella. When Gradle calls `useJUnitPlatform()`, both engines can coexist, but they share nothing beyond reporting infrastructure.

Quarkus's `QuarkusTestExtension` implements **seven Jupiter extension interfaces**: `BeforeAllCallback`, `AfterAllCallback`, `BeforeEachCallback`, `AfterEachCallback`, `InvocationInterceptor`, `ParameterResolver`, and `ExecutionCondition`. The most critical is `InvocationInterceptor`, which Quarkus uses to intercept the test class constructor and replace the JUnit-created instance with a **CDI-managed bean loaded through Quarkus's custom classloader**. This is where `@Inject` fields get populated, HTTP test resources get wired, and mock beans get installed. When Kotest runs a spec, none of these interceptor callbacks fire. The test instance is created by Kotest's own reflection-based instantiation (`javaReflectNewInstance`), completely outside Quarkus's CDI container. The result is the `UninitializedPropertyAccessException` that every developer hits when attempting `@Inject` in a Kotest spec.

Kotest does provide a `JUnitExtensionAdapter` in `kotest-extensions-junit5` that can wrap some Jupiter extensions into Kotest listeners. It supports `BeforeEachCallback`, `AfterEachCallback`, `BeforeAllCallback`, `AfterAllCallback`, and `TestInstancePostProcessor`. But it **does not support `InvocationInterceptor`, `TestInstanceFactory`, or `ParameterResolver`** — precisely the interfaces Quarkus depends on most. Even wrapping `QuarkusTestExtension` manually fails with null bean errors, because the adapter cannot replicate the constructor interception that bootstraps CDI.

## Three distinct blockers compound the problem

**Blocker 1: Test instance creation.** Quarkus needs to create test instances through its CDI container (`runningQuarkusApplication.instance(testClass)`) using classes loaded by `QuarkusClassLoader`. Kotest creates instances via its own `ConstructorExtension` chain using standard reflection. These two instance creation paths are mutually exclusive without explicit bridging code.

**Blocker 2: Classloader isolation.** Quarkus performs build-time augmentation (bytecode transformations for Panache, CDI interceptors, REST endpoint discovery) that is only visible to classes loaded by `QuarkusClassLoader`. Since Quarkus 3.22 (March 2025), a `FacadeClassLoader` hooks into JUnit Platform's `LauncherInterceptor` SPI during the discovery phase to route class loading. Test classes must be loaded by this classloader to see augmented bytecodes and participate in CDI. Kotest's ClassGraph-based discovery scans the standard classpath, bypassing this entirely.

**Blocker 3: DevMode test discovery.** Quarkus's continuous testing runner (`JunitTestRunner.collectTestAnnotations()`) hardcodes lookups for JUnit annotations — `@Test`, `@RepeatedTest`, `@ParameterizedTest`, `@TestFactory`, `@TestTemplate`. The source code contains a telling comment: *"this is not 100% correct, discovery needs to be based on class name. We can fix it when someone complains."* Kotest specs, which are discovered as `Spec` subclasses rather than annotated methods, are completely invisible to `quarkus:dev` continuous testing.

## Every attempted approach has failed

**The JUnitExtensionAdapter approach** (tried in `kotest-examples-quarkus`): Wrapping `QuarkusTestExtension` in Kotest's adapter and registering it as a Kotest extension. Failed because the adapter cannot bridge `InvocationInterceptor` or `TestInstanceFactory`, so CDI injection never activates.

**The AnnotationSpec approach** (tried by GavinRay97 in quarkus#19749): Using Kotest's `AnnotationSpec` — which mimics JUnit's method-based `@Test` annotations — with `@QuarkusTest`. Failed for the same root cause: `AnnotationSpec` still runs under Kotest's engine, not Jupiter's, so `@ExtendWith(QuarkusTestExtension.class)` is never processed.

**The @TestFactory/DynamicTest approach** (proposed but never implemented): Inspired by `quarkus-cucumber`, this would wrap Kotest specs inside a JUnit `@QuarkusTest` class that produces `DynamicNode` instances. Never built, and likely problematic — Quarkus has known issues with `@Inject` not working reliably in `@TestFactory` tests (quarkus#10435, #46473).

The Kotest maintainer (sksamuel) assessed the situation directly: *"The quarkus junit stuff seems to do a lot of magic classpathy stuff. What they really need to do is make it test framework agnostic."* He contrasted it with Spring: *"Spring keeps it well abstracted, and so it's easy(ish) to plug into Spring. They have their own before/after test interfaces, so junit and kotest can both map to that."*

## How Spock solved the same problem — and what Kotest can learn

The Spock framework faced an identical challenge: it implements its own JUnit Platform test engine and doesn't process Jupiter's `@ExtendWith`. The Spock community produced three independent solutions, each instructive.

**roggenbrot/quarkus-spock** built a custom Spock extension (`IAnnotationDrivenExtension`) that manually maps Spock lifecycle methods to the corresponding `QuarkusTestExtension` interceptor methods — `setupSpec` → `beforeAll`, `setup` → `beforeEach`, and so on. This "delegation bridge" invokes Quarkus's internal methods directly, effectively reimplementing the Jupiter extension contract in Spock's terms.

**RapidTransit/quarkus-spock** took a different route: a proper Quarkus extension (with runtime and deployment modules) that bootstraps Quarkus independently using a custom `@QuarkusSpec` annotation, bypassing `QuarkusTestExtension` entirely.

**xvik/spock-junit5** (the most mature, published on Maven Central) implements general JUnit 5 extension support as a global Spock extension. It auto-detects `@ExtendWith` annotations and wires them into Spock's lifecycle, creating compatible `ExtensionContext` implementations. Even this library explicitly notes that `TestInstanceFactory` extensions are **not supported** because Spock manages its own test instances.

The key lesson: all three approaches required **deep lifecycle mapping and significant engineering effort**. None is a trivial wrapper. And all remain somewhat fragile — the Quarkus 3.22 classloading rewrite (which moved from `TestInstanceFactory` to `InvocationInterceptor.interceptTestClassConstructor`) would have broken any integration that relied on the previous internal API.

## What a working integration would actually require

There are three realistic paths, each with different tradeoffs and owners.

**Path A — A dedicated Kotest extension (third-party or Kotest-maintained).** This would implement Kotest's `ConstructorExtension` to create spec instances through Quarkus's CDI container, `PostInstantiationExtension` for field injection, and lifecycle listeners (`BeforeSpecListener`, `AfterSpecListener`, `BeforeTestListener`, `AfterTestListener`) to manage application startup/shutdown. It would need to directly invoke Quarkus's internal bootstrap APIs — `QuarkusTestExtension`'s methods for augmentation, classloader setup, and CDI container initialization — rather than going through Jupiter. This is analogous to how `kotest-extensions-spring` uses Spring's `TestContextManager` directly instead of Spring's Jupiter extension. The challenge is that **Quarkus exposes no stable, framework-agnostic test API** equivalent to `TestContextManager`. The extension would depend on Quarkus internals that change across versions (as the 3.22 rewrite demonstrated).

**Path B — A proper Quarkus extension in Quarkiverse.** Following RapidTransit's Spock model, this would be a Quarkus extension with deployment and runtime modules. It would define a `@QuarkusKotestTest` annotation (or similar), register Kotest specs with Quarkus's build-time processing, and handle classloader setup. This has the advantage of using Quarkus's extension SPI properly but requires someone deeply familiar with both Quarkus extension development and Kotest internals.

**Path C — Quarkus abstracts its test infrastructure.** The cleanest long-term solution. Quarkus would extract a framework-agnostic test lifecycle API (similar to Spring's `TestContext` framework) that JUnit Jupiter, Kotest, Spock, and other frameworks could all target. The existing `QuarkusTestBeforeClassCallback` / `QuarkusTestAfterConstructCallback` SPI (loaded via `ServiceLoader`) is a step in this direction but doesn't cover the critical instance creation and classloader management. This would require Quarkus core team buy-in, and the open issue (#19749) has had **no assignee, no PRs, and no milestone** since August 2021.

## Current state and practical recommendations

As of Quarkus 3.31.x and Kotlin 2.3.0, the situation is unchanged. The `kotest-examples-quarkus` repository was archived in February 2025, signaling deprioritization. Quarkus's 3.22 classloading rewrite (which eliminated the serialization boundary between test and runtime classloaders) theoretically simplifies future integration — the blog post mentions improved support for "test frameworks such as Pact" — but no Kotest-specific work has materialized. There is no Quarkiverse Kotest extension, no community library on Maven Central, and no active PRs on either project.

The practical workarounds Kotlin developers use today are straightforward. Most teams adopt a **split testing strategy**: Kotest for pure unit tests and business logic (where Quarkus's container isn't needed), and JUnit 5 with `@QuarkusTest` for integration tests requiring CDI injection. A lighter variant uses JUnit 5 as the test runner but imports **Kotest's assertion library** (`kotest-assertions-core`) for its expressive matchers — `shouldBe`, `shouldContain`, `shouldThrow` — which work perfectly inside JUnit test methods. For end-to-end testing, some teams run Kotest specs that test Quarkus services as external HTTP clients, bypassing the `@QuarkusTest` machinery entirely.

## Conclusion

The Kotest-Quarkus incompatibility is not a bug to be fixed with a small PR — it's a **structural mismatch** between two frameworks that made fundamentally different architectural decisions about test lifecycle ownership. Kotest chose to be an independent JUnit Platform engine with its own extension model. Quarkus chose to deeply leverage Jupiter's extension points, particularly `InvocationInterceptor`, for CDI integration and classloader management. Bridging this gap requires either Quarkus providing a framework-agnostic test API (unlikely without core team prioritization), or someone building and maintaining a substantial integration layer that tracks Quarkus's internal API changes across versions. The Spock community's solutions prove it's technically feasible but engineering-intensive. Until someone invests that effort, the split testing strategy — Kotest for unit tests, JUnit 5 for `@QuarkusTest` integration tests — remains the pragmatic choice for Kotlin teams on Quarkus.
